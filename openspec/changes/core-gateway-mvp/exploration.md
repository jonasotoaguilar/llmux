Establish the smallest vertical slice that turns `docs/foundation` into a runnable,
reviewable backend. It must (1) install real backend tooling, (2) turn on strict
TDD, (3) expose an OpenAI-compatible health + chat-completions boundary, and

PR: two providers, full router with fallback, API-key auth against PostgreSQL,

on `docs/foundation` rather than `main` because PR #2 is blocked on external

the foundation to merge. This is an orchestrator-side decision, not a

## 5. Recommendation

### 5.1 Dependency manager: **`uv`**

| Dimension | `uv` | `poetry` | `pip-tools` |
|-----------|------|----------|-------------|
| Speed | Fast (Rust, parallel resolver) | Slow | Medium |
| Lockfile | `uv.lock` (cross-platform) | `poetry.lock` | `requirements.txt` + `.txt` hashes |
| Tooling surface | Runs scripts, manages Python versions, virtualenvs | Just deps + venv | Just deps + venv |
| Matches user AGENTS.md | Yes | No | Partial |
| Maturity for CI | Good; `uv sync --frozen` is one-shot | Good; `poetry install --no-interaction` | Standard pip |
| Project-config nesting | `[tool.uv]` in `pyproject.toml` | `[tool.poetry]` in `pyproject.toml` | Out-of-band `requirements.in` |

**Pick `uv`.** It is consistent with the user's global config, gives a single
`uv.lock` that doubles as the contract for CI, and avoids a second decision
(virtualenv handling) by managing that too. The 3-line reviewer mental model
("`uv sync` to install, `uv run pytest` to test, `uv lock --upgrade` to
refresh") is easier to onboard than poetry's. The lockfile is checked in,
so CI is reproducible without depending on PyPI registry state at build time.

Trade-off accepted: `uv` is younger than poetry, so a fallback line in
`CONTRIBUTING.md` ("if you don't have uv, see `docs/dev/setup.md`") may be
worth adding — but that is documentation, not code, and is in-scope for the
follow-up slice that adds `docker-compose.yml`.

## 8. Risks

- **Review budget pressure.** §5.2 estimates ~370 lines. A reviewer asking
  for one extra test case per behavior plus a `.gitignore` push it over.
  Mitigation: drop `Dockerfile` first (it is MEDIUM severity, not BLOCKER),
  then drop the `docs/adr/*.md` status flips, then drop the
  `pytest.ini` block in favor of pyproject's `[tool.pytest.ini_options]`.
  Last to drop: the OTel stub (it can be a one-liner in `main.py`).
- **ADRs marked Proposed, not Accepted.** ADR-0001's `Status: Proposed` is
  technically honest today — no code reflects it. The moment slice 1
  lands, both ADRs are de-facto accepted. Slice 1 should flip the status
  to keep the docs honest. Risk if not done: a future reviewer can
  challenge the architecture mid-Phase-1, claiming the ADRs are still
  proposals.
- **PR #2 still open.** CI on this branch is fine (it runs on the branch
  itself), but anyone reviewing slice 1 will see `.env.example` and
  `Dockerfile` show up in a docs-only foundation PR's audit findings.
  This is fine — it is partial progress on Issue #1, not a regression —
  but the PR description should call that out explicitly.
- **Strict TDD is being turned on by a slice that has no production code
  to test.** The `strict_tdd: true` flip in `openspec/config.yaml` is
  semantically correct (the test runner exists, CI executes it) but
  philosophically under-tested (the only "production" code is a 501
  stub). The orchestrator should phrase the proposal as "TDD capability
  established" rather than "TDD adopted" — the second slice with real
  provider code is the first true TDD consumer.
- **`uv` lockfile reviewability.** First-time reviewers may not know the
  `uv.lock` format. This is a one-time cost; the `CONTRIBUTING.md` delta
  in this slice names the three commands they need (`uv sync`,
  `uv run pytest`, `uv run uvicorn llmux.main:app`).
